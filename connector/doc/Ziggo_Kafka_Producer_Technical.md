# Ziggo Kafka Producer

## About

The Ziggo Kafka Producer is a virtual DataMiner connector that reads DataMiner trending offload CSV files from shared network folders and publishes each data point as an enriched JSON message to an Apache Kafka topic. Every message is enriched with live element metadata (element name, view location, protocol name, element type, element state, parameter unit, and resolved discrete values) sourced from a local DMS metadata cache.

No Stream Viewer traffic is generated by this connector. All communication happens via the Confluent Kafka client (librdkafka) and direct file system access to the configured shares.

## Configuration

### Connections

#### Virtual Connection - Main

This connector uses a virtual connection and does not require any input during element creation.

### Initialization

After creating the element, configure the following settings before the connector can start producing messages.

- **Authentication** — on the **General** page:

  - **Username**: SASL/PLAIN username for authenticating against the Kafka broker.
  - **Password**: SASL/PLAIN password.
  - **CA Certificate Location**: Full local path to the CA certificate file (e.g., `C:\Certs\ca.pem`). When filled in, the security protocol upgrades from `SASL_PLAINTEXT` to `SASL_SSL`. Leave empty if SSL/TLS is not required.

  > [!IMPORTANT]
  > Changing any authentication setting causes the Kafka producer to be rebuilt immediately. The current producer is flushed before it is disposed.

- **Shared folders** — on the **Offload File Settings** subpage (for Failover Agents):

  - **Shared Folder 1**: UNC path to the primary folder containing offload CSV files (e.g., `\\hostname\offload`).
  - **Shared Folder 2**: Optional. UNC path to a fallback folder. Leave empty if redundancy is not needed.
  - **Shared Folder Username**: Account used to connect to the shares. Must have **read and write** access, as the connector deletes files after successful delivery.
  - **Shared Folder Password**: Password for the shared folder account.

- **Kafka brokers and topic** — on the **Kafka Broker Settings** subpage:

  1. Set the **Kafka Topic** to the topic name that messages should be published to.
  1. Right-click the **Kafka Brokers** table and select **Add Item** to add each broker endpoint (by adding **IP Address** and **Port** — default *9092*).

     Add all broker addresses for a highly available cluster.

## How to Use

### Offload directories

Every 10 seconds, the connector scans the configured shared folders and discovers subdirectories named exactly `Data` or `Data_Snapshot`. These appear automatically in the **Offload Directories** table on the **Offload File Settings** subpage. New rows default to **Discover Files: Disabled**. Toggle this to *Enabled* per directory to activate publishing.

Use the **Time Interval** column (minimum 5 s) to control how often each directory is checked for new files. The **Files to Process** column provides a live backlog count and has alarm thresholds configured (Warning ≥ 10 up to Critical ≥ 1 000) so that a growing backlog raises an alarm.

### Data flow

When a directory's time interval elapses, all CSV files found in it are queued. The connector processes one file at a time:

1. Each CSV line is parsed against the [DataMiner trending offload database structure](https://aka.dataminer.services/structure-offload-database) and enriched with metadata from the local DMS cache.
1. The resulting JSON message is produced to the configured Kafka topic. The Kafka message **key** is set to `DmaId/ElementId/ParameterId/DisplayKey`, enabling key-based partition ordering on the consumer side.
1. After all messages for a file are flushed and confirmed by the broker, the file is deleted from the share. If delivery errors are returned, the file is left on the share for reprocessing.

The JSON structure of each message is shown below:

```json
{
  "DMA_EID_PID_DISPLAYKEY": "101/5/100/row1",
  "DMA_EID_PID": "101/5/100",
  "ElementName": "My Element",
  "ParamName": "CPU Load",
  "DisplayKey": "row1",
  "Location": "Belgium/Brussels/Rack A",
  "Protocol": "Generic Ping",
  "ElementType": "Server",
  "ElementState": "Active",
  "DataPoint": {
  "Status": 0,
  "Unit": "%",
  "Type": "double",
  "Value": 42.5,
  "DisplayValue": "42.5 %",
  "TimeStampUnix": 1714204800
  }
}
```

| Field | Description |
|--|--|
| `DMA_EID_PID_DISPLAYKEY` | Unique data point identifier (`DmaId/ElementId/ParameterId/DisplayKey`). Also used as the Kafka message key. The display key suffix is omitted for scalar parameters. |
| `DMA_EID_PID` | Parameter-level identifier without the display key. |
| `ElementName` | Element name as recorded in the offload file. |
| `ParamName` | Parameter name as recorded in the offload file. |
| `DisplayKey` | Table row display key; empty for scalar parameters. |
| `Location` | DataMiner view path of the element. |
| `Protocol` | Connector name of the element. |
| `ElementType` | Element type string from the protocol definition. |
| `ElementState` | Current operational state of the element (*Active*, *Paused*, etc.). |
| `DataPoint.Status` | Trending status flag (0 = normal value). |
| `DataPoint.Unit` | Physical unit of the parameter (e.g., `%`, `dBm`). |
| `DataPoint.Type` | Parameter data type: `double`, `string`, or `discreet`. |
| `DataPoint.Value` | Measured value. Discrete values are resolved to their display string. |
| `DataPoint.DisplayValue` | Human-readable value including unit (e.g., `42.5 %`). |
| `DataPoint.TimeStampUnix` | Measurement timestamp as a Unix epoch in seconds (UTC). |

### Error handling and the Failed directory

**Broker Connectivity Status** on the **General** page reflects whether the last flush succeeded. If it shows *Not Connected*, check the DataMiner logs for delivery error details.

When a line cannot be parsed, **Parsing Lines Errors** is incremented, the line is skipped, and the remaining lines in the file are still processed. When a whole file cannot be processed (e.g., because the internal producer queue is full), **Processing Files Errors** is incremented. In both cases, the problematic content is moved to a **Failed** subdirectory alongside the source directory (e.g., `\\server\offload\Data\Failed\`).

To recover, resolve the root cause, move the files from **Failed** back to **Data**, and click **Clear Errors** to reset the counters.
